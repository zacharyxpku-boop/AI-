# wenai 产品状态

更新时间: 2026-05-04

## 当前判断

wenai 已经从泛 AI 工具集合推进到 **10 SKU POC 子站** 的产品形态。现在的核心闭环是:

访客看定位与案例 -> 提交询盘 -> 生成 POC 标准包 -> 跑对应 Pipeline -> 进入后台项目板 -> 复盘 -> 推进主站合同/支付。

## 已经做硬的部分

1. **定位**
   - 明确不做低价自助 AI 工具箱。
   - 主目标是高意向客户的 10 SKU POC 承接与合同推进。

2. **客户侧**
   - `/poc` 有 POC 说明、交付目录、边界和匿名证据。
   - `/cases` 与 `/cases/[slug]` 有可分享案例详情。
   - `/modules/standard-pack` 可以生成完整交付包、老板验收摘要、内部执行 brief、客户跟进话术。
   - 标准包会推荐主执行 Pipeline 和辅助 Pipeline, 客户不用自己猜下一步。

3. **交付侧**
   - SOP workflow 已代码化, 覆盖 benchmark、Podcast UGC、街采、Slideshow、批量 UGC、Animated Ads、剪辑优化。
   - 上新、主图、视频、AB 测试、达人外联、数据洞察等 Pipeline 已接标准包入口。

4. **运营侧**
   - `/admin/inquiries` 有线索评分、验收准备、合同准备、复盘决策、合同下一动作和活动时间线。
   - `/admin/metrics` 看合格率、交付率、复盘率、合同率、沉默项目和时间线覆盖。

5. **验证**
   - `scripts/verify.ps1` 覆盖重点测试、TypeScript、ESLint 和 Next build。

## 仍然不足

1. **真实外部配置未接**
   - API key、模型 token、支付、邮件发送和线上域名需要上架阶段配置。

2. **客户自助仍偏 POC**
   - 标准包已经能自助生成, 但完整 Pipeline 跑完后的自动验收报告还没有在每条 Pipeline 全覆盖。

3. **真实案例证据不足**
   - 当前案例是匿名/演示结构, 可以解释交付逻辑, 但不能替代真实客户背书。

4. **长期数据壁垒还没厚**
   - 需要真实 SKU、复盘结果、合同结果持续进入系统, 才能形成不可复制的数据飞轮。

## 下一批优先级

1. 把关键 Pipeline 的结果统一生成验收报告和复盘包。
2. 给客户侧加“POC 启动清单”, 让客户提交前能自检资料完整度。
3. 上线后沉淀真实案例, 替换演示证据。
4. 接主站合同/支付后的状态回流, 让合同结果进入 metrics。
